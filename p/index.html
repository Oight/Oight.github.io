阅读时长: 6 分钟

你现在这个题的结构（从你前面给出的行为来看）属于 “主程序在 x86_64 上跑，但把关键校验逻辑塞进另一段小代码里，用 Unicorn 模拟另一种架构去跑” 的典型逆向题。对新手来说，一开始会很懵：因为你在 IDA/Ghidra 里看 main 逻辑很短，但就是找不到真正的校验算法 —— 它藏在“被模拟执行”的那段代码里。

1. 新手逆向的正确打开方式：先动态“看它想干嘛”

第一步别急着上反编译，先运行：

1
2

chmod +x vvvmmm
./vvvmmm

你要观察 3 件事：

  1. 它要你输入什么（提示里是否写着 %48c / 48 / flag 之类）
  2. 输入长度是不是固定（很多题就是固定 32/40/48 字节）
  3. 成功/失败的分支是什么输出（比如 Good. / Try again）

结论：你得先知道“验证的输入是什么形态”。比如 %48c 意味着“精确读取 48 个字符”，包含空格也算字符（这和 %s 不一样）。

2. 静态初筛：先识别“壳/加壳/保护”

对二进制先做体检：

1
2
3
4

file vvvmmm
checksec --file=vvvmmm
strings -a vvvmmm | head
strings -a vvvmmm | grep -i upx

如果你看到：

  • UPX!、或者 strings 里有很多 UPX 特征
  • file 显示 “packed” 或者体积很小但跑起来又很复杂

那多半是 UPX 壳。对新手来说，UPX 是最好处理的壳：

1
2

upx -t vvvmmm        # 测试是不是UPX
upx -d vvvmmm -o vvvmmm.unpacked

后面所有分析都优先用 vvvmmm.unpacked。

为什么必须先脱壳？ 因为没脱壳时，main 逻辑/字符串/代码结构都被压缩重排了，你在反编译器里看到的会很“假”，很难跟踪。

3. 在 Ghidra/IDA 里快速定位 main 和输入点

新手最容易迷路，所以推荐你用“字符串定位法”：

  1. 打开 Ghidra / IDA
  2. 搜索字符串（Strings window）：
    • 找输入提示（比如 input、%48c、Try again、Good）
  3. 对该字符串点 Xref（交叉引用）跳到使用该字符串的位置

一般你会很快落到 main 附近，看到类似：

  • printf("...input %48c>")
  • scanf("%48c", buf) 或 read(0, buf, 48)

这一步你就拿到了：

  • 输入长度
  • 输入缓冲区地址/变量
  • 成功/失败输出位置

4. 发现“关键点”：Unicorn 模拟器调用链

这种题最关键的特征是，你会在 main 附近看到一堆函数调用名字很像：

  • uc_open
  • uc_mem_map
  • uc_mem_write
  • uc_reg_write
  • uc_emu_start
  • uc_reg_read
  • uc_close

只要看到这些，几乎可以确定：

✅ 真正的校验逻辑不是 x86_64 指令，而是在某段 bytes 里，用 Unicorn 模拟某个架构执行。

你接下来要做的是把下面这些信息从 main 里“抄下来”：

4.1 它模拟的是什么架构？

在 uc_open(arch, mode, &handle) 里就能看见：

  • UC_ARCH_RISCV / UC_ARCH_ARM / UC_ARCH_MIPS 等
  • UC_MODE_RISCV64 / UC_MODE_ARM / UC_MODE_64 等

4.2 它把“模拟代码”写到哪里了？

你会看到类似：

  • uc_mem_map(0x0, 0x1000, perms)
  • uc_mem_write(0x0, code_blob, code_len)

这说明：

  • 模拟执行的“代码段”从地址 0x0 开始
  • code_len 可能是 0x296 这种固定长度

4.3 输入数据放在模拟器哪段内存？

一般会有：

  • uc_mem_map(0x10000000, 0x1000, perms)
  • uc_mem_write(0x10000000, input, 48)
  • uc_reg_write(A0, 0x10000000)

这说明：

  • 模拟代码把 A0 当作输入指针（RISC-V 的 a0 是第一个参数寄存器）
  • 输入在模拟内存 0x10000000

4.4 它还有没有第二个参数/常量字符串？

常见还有：

  • uc_mem_write(0x10001000, "xxx...", 32)
  • uc_reg_write(A1, 0x10001000)

这说明模拟代码会用常量串参与生成 key/seed。

4.5 成功/失败怎么回传？

你会看到类似：

  • uc_emu_start(start=0, end=code_len, ...)
  • uc_reg_read(A0, &ret)
  • if (ret) success else fail

✅ 这就非常清晰了：模拟代码返回值放在 a0。 （RISC-V 调用约定：a0 既是第一个参数，也通常用于返回值）

5. 真正逆向从这里开始：把那段“RISC-V 小代码”提出来

你要做两件事：

5.1 提取 code blob（bytes）

你有两种简单办法：

办法 A：在 Ghidra/IDA 里直接导出

  • 找到 uc_mem_write(0x0, <指针>, code_len) 的 <指针>
  • 跳到该地址，通常在 .rodata 或某个全局数组里
  • 选中 code_len 字节，导出为 rv.bin

办法 B：用 gdb 在运行时 dump（对新手也很友好） 在 uc_mem_write 前下断点，查看它第二个参数指向的内存，dump 出来。

这一步的意义： 你从“反编译 C”切换到“反汇编另一种架构”，题目核心就在这段 rv.bin 里。

6. 反汇编 RISC-V（或目标架构）代码，让它变成你能读懂的逻辑

如果是 RISC-V 64，你可以用：

1

riscv64-linux-gnu-objdump -D -b binary -m riscv:rv64 rv.bin > rv.asm

（不同系统工具链名字可能不一样，关键是 objdump 能按 riscv binary 解）

然后你重点看三块：

  1. 输入处理循环：通常会从 a0 指向的 buffer 读 48 字节
  2. key/seed 生成：可能会用 a1 指向的常量字符串做 hash/seed
  3. 比较常量：最终一定会有一段 if (computed != expected) fail 的逻辑

7. 新手最推荐的解法：不用完全看懂汇编，也能把 flag 反推出来

你前面描述的这种结构（XOR/流加密/比较常量）通常具备一个特性：

如果校验过程是 out[i] = in[i] XOR keystream[i] 并拿 out 去跟 expected 比较 那么： in = expected XOR keystream

所以你只需要搞到两样东西：

  • expected：比较用的 48 字节常量
  • keystream：程序生成的 48 字节密钥流

7.1 expected 怎么拿？

两种可能：

  • 常量以“表”的形式存在（48 字节或 12 个 32-bit word）
  • 常量被拆成若干 lui/addi / li 拼出来

你可以：

  • 在汇编里找到“逐字比较”的地方，逆着往上找常量来源
  • 或者更简单：写 Unicorn hook，在比较点把 expected 读出来（更适合不爱看汇编的人）

7.2 keystream 怎么拿（超实用技巧：全 0 输入）

如果 out = in XOR keystream，那么当你让 in = 0：

  • out = 0 XOR keystream = keystream

也就是说： ✅ 用全 0 的 48 字节输入跑一遍模拟器，然后读取被变换后的 buffer，就能直接得到 keystream。

然后你再用 expected XOR keystream 就能恢复正确输入。

这个技巧对新手特别友好： 你不用完全理解 keystream 怎么生成的，只要能“跑起来并读内存”就够了。

from unicorn import Uc, UC_ARCH_RISCV, UC_MODE_RISCV64, UC_PROT_ALL
from unicorn.riscv_const import UC_RISCV_REG_A0, UC_RISCV_REG_A1, UC_RISCV_REG_SP

CODE_ADDR = 0x0
CODE_LEN  = 0x296            # 以你在 main 里看到的为准

IN_ADDR   = 0x10000000
IN_LEN    = 48

STR_ADDR  = 0x10001000
CONST_STR = b"e4Y8YRXVzg2HRrCUy35CM0Txq91HzMGZ\x00"  # 以 main 里看到的为准

STACK_ADDR = 0x70000000
STACK_SIZE = 0x10000
SP_INIT    = STACK_ADDR + STACK_SIZE - 0x100

with open("rv.bin", "rb") as f:
    code = f.read()

uc = Uc(UC_ARCH_RISCV, UC_MODE_RISCV64)

# map code
uc.mem_map(CODE_ADDR, 0x1000, UC_PROT_ALL)
uc.mem_write(CODE_ADDR, code[:CODE_LEN])

# map input
uc.mem_map(IN_ADDR, 0x1000, UC_PROT_ALL)
uc.mem_write(IN_ADDR, b"\x00" * IN_LEN)  # 全0输入拿keystream

# map const string
uc.mem_map(STR_ADDR, 0x1000, UC_PROT_ALL)
uc.mem_write(STR_ADDR, CONST_STR)

# map stack
uc.mem_map(STACK_ADDR, STACK_SIZE, UC_PROT_ALL)
uc.reg_write(UC_RISCV_REG_SP, SP_INIT)

# set args
uc.reg_write(UC_RISCV_REG_A0, IN_ADDR)
uc.reg_write(UC_RISCV_REG_A1, STR_ADDR)

# run
uc.emu_start(CODE_ADDR, CODE_ADDR + CODE_LEN)

# 读取变换后的输入区：这就是 keystream（在“XOR型”题里）
keystream = uc.mem_read(IN_ADDR, IN_LEN)
print("keystream =", keystream.hex())

接下来你拿到 expected 后：

1
2

inp = bytes([expected[i] ^ keystream[i] for i in range(48)])
print(inp)

如果 inp 是可打印字符，你就能直接看到输入字符串；如果有不可打印，也可以用 repr 或 base64 输出。

9. 你最终要明白的“题目套路总结”（这题的核心思想）

这类题想考你 3 件事：

  1. 识别壳/脱壳（UPX 是入门必会）
  2. 识别“模拟执行”（看到 uc_* 就要警觉：校验不在本机指令）
  3. 从“比较常量”反推输入（XOR/流加密/变换 → 可逆 → 求解输入）

新手只要抓住这条主线，不需要你一开始就能读懂所有汇编，也能做出来。
